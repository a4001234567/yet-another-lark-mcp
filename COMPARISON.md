# lark-mcp-server — 与同类工具的对比

## 是什么

通过 MCP 协议将飞书接入 AI 助手的个人工具。在 Claude Desktop、Cursor、Zed 等支持 MCP 的客户端中，可直接操作飞书日历、任务、消息和文档，无需切换应用。

---

## 与 feishu-openclaw-plugin 的对比

两者接入方式（MCP stdio）和认证方式（OAuth 设备流）相同。主要差别在于 API 的取舍与封装。

openclaw 插件追求完整覆盖，接近飞书原始 API 的参数粒度，适合需要精细控制的场景。lark-mcp-server 的取向相反：精简到个人常用场景，同时在内部处理好几个飞书 API 的已知边界问题：

- **日历查询去噪**：`instance_view` 在时间范围查询中，循环日程的模板即使没有实例落在查询段内也可能被命中。服务器内部交叉比对后，只返回指定时间段内真实存在的实例。
- **循环日程单次修改**：循环日程的单次实例在未被实例化之前无法单独修改，直接操作会影响所有后续场次。服务器在修改前自动触发实例化（`instance_start_time_admin`），再对指定场次进行操作。
- **任务归属**：创建任务时自动将调用者加为负责人，确保任务立即出现在「我的任务」中。

此外，将 openclaw 中其他 MCP 工具普遍欠缺的能力也一并实现：watch loop 实时消息监听、内置交互卡片（确认/表单/进度）、cron 定时任务——这也是实现对话体验的基础。

---

## 与直接调用飞书 API 的对比

主要解决三件事：Token 自动管理（OAuth 设备流授权 + 到期前自动刷新）、上述 API 边界问题的统一处理、无需自建 Webhook 服务器即可接收实时消息。

---

## 核心优势

**适配现有工具链。** 基于 MCP 协议，任何支持 MCP 的客户端开箱即用，不绑定特定平台。

**真正的对话体验。** watch loop + 实时消息监听 + 交互卡片，让飞书成为完整的对话界面：用户发消息，AI 实时响应、追问细节、发送确认卡片等待回应，再继续执行。不是单次指令执行，而是持续交互。

**业务逻辑封装到位。** 飞书 API 的若干边界问题在服务器层统一处理，调用方无需了解背后的实现细节。
